## Issue Investigation: Oversized Actor Exclusion

### Initial Problem
- User reported skysphere being selected despite being far outside tiny selection volume
- Skysphere extends across entire level but pivot point was inside small volume at origin
- Expected: Massive actors like skyspheres should be excluded from selection

### Implementation Attempt
Added "Exclude oversized actors" feature with:
- Checkbox to enable/disable feature
- Slider for threshold multiplier (1x-10x)
- Volume size comparison logic
- JSON persistence for settings

**Files Modified:**
- `VolumeSelectionSettings.h` - Added `bExcludeOversizedActors` and `OversizedActorThreshold` fields
- `VolumeSelectionSettings.cpp` - Added JSON serialization for new settings
- `OrganisationEdModeToolkit.cpp` - Added UI checkbox and slider controls
- `SelectActorsInVolume.cpp` - Added size comparison logic in `PassesVolumeFilter()`

### Debugging Process

**Round 1: Volume Bounds Investigation**
- Volume bounds reported as 100,000 x 100,000 x 4,000 cm instead of expected 500 x 500 x 20 cm
- Added debug logging to investigate bounds calculation
- Tested multiple bounds retrieval methods:
  1. `GetComponentsBoundingBox(true)` - returned inflated values
  2. `BrushComponent->Bounds.GetBox()` - returned 0x0x0 for REVSelectionVolume
  3. `BoxComponent->Bounds.GetBox()` - returned inflated values
  4. `GetScaledBoxExtent()` - still inflated due to actor scale

**Round 2: Scale Analysis**
```
Debug Output:
  Unscaled Extent: 100.00 x 100.00 x 100.00 (from constructor)
  Scaled Extent: 50,000.00 x 50,000.00 x 2,000.00
  Component Scale: 500.00 x 500.00 x 20.00
  Actor Scale: 500.00 x 500.00 x 20.00
```

Initial assumption: Volume was incorrectly scaled too large (500 x 500 x 20 instead of 5 x 5 x 0.2)

**Round 3: Visual Inspection**
User provided screenshots showing:
- Volume visually correct size (~500x500x20 cm, matching 5 grid squares)
- Actor scale confirmed as `(X=500.000000, Y=500.000000, Z=20.000000)` via copy/paste
- Skysphere visually enormous (filling entire viewport)
- Test cube scaled to `(X=20000, Y=20000, Z=20000)` to match skysphere size

### Root Cause Discovery

**The Fundamental Issue:**
```
Skysphere Properties:
  Actor Scale: (X=400, Y=400, Z=400)
  Collision Bounds: 400 x 400 x 400 cm (reported by GetComponentsBoundingBox)
  Visual Mesh Bounds: ~20,000 x 20,000 x 20,000 cm (actual visible size)
```

**Key Finding:** Environmental actors (skyspheres, skyboxes, atmospheric fog) intentionally have tiny collision bounds that don't match their visual mesh size. This prevents them from interfering with gameplay physics and collision detection.

**Why the Feature Can't Work:**
- `GetComponentsBoundingBox()` returns collision/physics bounds, not visual bounds
- No reliable API to get actual visual mesh extents for all actor types
- Skysphere appears as a small 400cm actor to the comparison logic
- Volume at 100,000cm is reported as 250x larger than skysphere
- Feature logic inverted: meant to exclude large actors, instead excludes small actors

## Resolution

### Decision
Removed "Exclude oversized actors" feature entirely due to:
1. Unreliable bounds reporting for environmental actors
2. No way to distinguish collision bounds from visual bounds generically
3. Existing solutions (Exclude Classes, Exclude Tags) already handle this use case
4. Feature created false sense of functionality while not working as intended

### Alternative Solutions Provided
**Method 1: Exclude by Class**
- Use "Exclude Actor Classes" dropdown
- Select `StaticMeshActor` or specific skysphere blueprint class
- Prevents skysphere from being selected regardless of position

**Method 2: Exclude by Tag**
- Add custom tag to skysphere (e.g., "EnvironmentActor")
- Add tag to "Exclude Tags" list in toolkit
- Reusable for all environmental actors that should never be selected

## Technical Insights Gained

### Unreal Engine Bounds System
1. **Collision Bounds vs Visual Bounds:**
   - `GetComponentsBoundingBox()` returns physics/collision bounds
   - Visual mesh extents may be orders of magnitude larger
   - No universal API to retrieve visual bounds for all actor types

2. **BoxComponent Scale Behavior:**
   - `GetUnscaledBoxExtent()` returns constructor extent (100x100x100)
   - `GetScaledBoxExtent()` multiplies by component scale
   - Actor scale compounds with component scale
   - Result: 100 * 500 = 50,000 per axis, * 2 for full box = 100,000

3. **Environmental Actor Design:**
   - Skyspheres, skyboxes, fog volumes intentionally have minimal collision
   - Prevents physics interference during gameplay
   - Visual mesh extends far beyond collision bounds
   - This is by design, not a bug

### Best Practices Learned
- Always verify assumptions with visual inspection + debug output
- Test edge cases (environmental actors) when building generic solutions
- Consider whether existing features (tags, class filters) already solve the problem
- Remove features that don't work reliably rather than shipping half-working solutions

## Notes for Future Development

1. **Bounds APIs Limitations:**
   - Don't assume `GetComponentsBoundingBox()` represents visual size
   - Environmental actors are special cases
   - Consider asking users to tag special actors rather than auto-detecting

2. **UI Philosophy:**
   - Fewer, well-working features > many half-working features
   - Existing tag/class filters are flexible enough for edge cases
   - Don't add UI controls that give false confidence

3. **Debug Strategy:**
   - Visual inspection + debug logging both required
   - Test with actual level content (skyspheres, not just cubes)
   - Ask users to provide exact transform values via copy/paste
